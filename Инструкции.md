# Промпт реализации макета

Скопируй этот промпт в Codex, когда нужно реализовать экран по Figma.

```text
Реализуй экран по Figma максимально точно.

Источник макета:
<ВСТАВЬ FIGMA URL>

Главные правила:
1. Сначала прочитай AGENTS.md и Инструкции.md в проекте.
2. Сними Figma data и Code Connect по экрану:
   - получи design context выбранного node;
   - проверь Code Connect / docs / Git links у ключевых Figma-компонентов;
   - проверь componentProperties, variantProperties, swaps и text properties;
   - если root-контекст слишком сжатый, используй Figma Plugin API read-only, чтобы снять дерево слоев и geometry программно.
3. Сопоставь geometry map с токенами и API DS:
   - figma px -> token variable -> class;
   - Figma component/property -> borrom-ds-test component/prop;
   - Figma state/variant -> code prop/state;
   - slot/content/action -> соответствующий slot или page-level composition.
4. Не верстай по PNG. PNG/screenshot из Figma используй только для визуального сравнения результата: похожи ли компоненты, отступы, пропорции, состояния и общая композиция.
5. Верстку строй по данным Figma: layers/nodes, auto-layout, constraints, variables, variants, text styles и properties.
6. Если у Figma-компонента есть Git link, Code Connect или docs, используй готовый компонент из borrom-ds-test. Не создавай локальный дубль компонента.
7. Не меняй исходники компонентов дизайн-системы ради одного экрана. Если компонент корректно работает в playground/Storybook, проблему ищи в page-level композиции, props, слотах, теме, контейнерах, overflow/z-index и размерах вокруг компонента.
8. Точные Figma component properties/variants/swaps переноси в props DS-компонента. Не угадывай size/type/color/iconOnly по картинке, если Figma уже отдала явное значение.
9. Если готового компонента нет, зафиксируй blocker/assumption и реализуй fallback по слоям Figma через MCP или Figma Plugin API read-only, не по PNG.
10. Если готовый DS-компонент не совпадает с конкретной экранной композицией, не переписывай компонент. Используй page-level fallback по Figma layers, сохраняя DS controls и tokens, и зафиксируй Temporary mapping.
11. Все цвета, отступы, радиусы, типографику и размеры бери из токенов проекта/Figma. Raw px/hex используй только если токена нет, с пометкой Temporary mapping.
12. Иконки бери из lucide-react.
13. Сохраняй модель размера Figma: fill остается fill, hug остается hug, fixed используется только если fixed указан в Figma или зафиксирован как Temporary mapping.
14. Перед финалом запусти npm run lint, npm run build и npm run encoding:check, если скрипт есть.
15. Запусти npm run server:start и сделай parity-pass: сравни локальный экран с Figma screenshot/PNG-контекстом, но исправления вноси по Figma data и токенам, а не подбором на глаз.
16. Если создаешь отчет, в том же файле укажи симптом, причину, принятое решение, что изменено, как упредить повторение, assumptions / temporary mappings и результаты проверок.

Ожидаемый результат:
- рабочий React/Vite экран;
- использование компонентов borrom-ds-test там, где они есть;
- точное соответствие auto-layout, spacing, typography, colors и states;
- geometry map для ключевых размеров и отступов;
- список assumptions / temporary mappings, если что-то не совпало с токенами или API компонентов;
- отчет о проверках lint/build и visual parity.
```

# Главный принцип

Проект должен воспроизводить Figma через реальные компоненты дизайн-системы, токены и структуру слоев.

Мы не верстаем по PNG и не переносим raw-значения из инспектора как финальную реализацию. PNG/screenshot/PNG-контекст из Figma используется только для visual parity после того, как структура уже снята по нодам. По PNG можно сверять, так ли выглядят компоненты, похожи ли отступы, пропорции, состояния и общая композиция.

Источник верстки всегда один: данные Figma layers/nodes, auto-layout, constraints, variables, variants и properties.

# Preflight перед реализацией

Перед первым кодом зафиксируй базовые факты:

- `npm view borrom-ds-test version` возвращает доступную версию пакета.
- В `package.json` зависимость указана как обычный npm package: `"borrom-ds-test": "*"`, `latest` или конкретная версия.
- Импорты остаются без версии: `import { Button } from "borrom-ds-test";`.
- Если npm возвращает `404` или `ENEEDAUTH`, это `install/access blocker`, а не повод менять import на git/file/link workaround без решения пользователя.
- `react`, `react-dom`, `@types/react`, `@types/react-dom` совместимы с peer/runtime ожиданиями DS.
- Если после старта виден белый экран, сначала проверяем совместимость React/React DOM/types с установленной версией `borrom-ds-test`, затем очищаем Vite cache и пересобираем проект.
- `src/styles/fonts.css` подключен до `src/tokens/index.css`, `src/styles/global.css` и `borrom-ds-test/styles.css`.
- Тема экрана определена до верстки: `themes_dark` или `themes_light`.
- Для ключевых Figma-компонентов проверены Code Connect, docs и Git links.
- Снята geometry из Figma: ширины контейнеров, высоты блоков, gaps, padding, constraints, fill/hug.
- Составлена короткая geometry map по ключевым расстояниям: `figma px -> token variable -> class`.

Если менялись React, React DOM или DS-пакет, очисти Vite cache:

```powershell
Remove-Item -Recurse -Force node_modules/.vite
```

# Figma data: как пройти по ноде и ничего не потерять

Цель этого прохода - снять не картинку, а структуру экрана. Нельзя начинать верстать большой root по догадкам, если Figma context пришел слишком сжатым.

## Минимальный порядок снятия данных

1. Открой Figma URL и зафиксируй `fileKey`, `nodeId`, viewport, тему и название экрана.
2. Получи `get_design_context` для выбранной root-ноды.
3. Проверь, есть ли Code Connect, docs или Git links у ключевых компонентных инстансов.
4. Если root-контекст слишком сжат, используй read-only Figma Plugin API или metadata-проход для полного дерева.
5. Сними дерево слоев сверху вниз: root -> sections -> groups -> component instances -> text/assets.
6. Для каждого ключевого node запиши geometry, layout и mapping в таблицы.
7. Перед кодом составь geometry map и список DS-компонентов.

## Что фиксировать для каждого важного node

| Категория | Что снять |
| --- | --- |
| Identity | name, node id, type, component name, variant name |
| Geometry | x/y, width, height, min/max, absolute/relative position |
| Auto-layout | direction, gap, padding, alignment, wrap, distribution |
| Resize model | fill, hug, fixed, constraints |
| Visual style | fills, strokes, radius, shadows, opacity |
| Typography | text content, font family, size, weight, line-height, letter spacing, text style |
| Component props | `componentProperties`, `variantProperties`, boolean/text/instance swap props |
| Slots | left/right icons, actions, labels, helper text, children/content areas |
| States | disabled, error, active, selected, open, hover-like visible states |
| Assets | images, masks, crop behavior, object-fit, avatar/file/image sources |

## Детальный node-проход

Для root:

- проверь размер экрана и responsive ожидание;
- проверь тему (`themes_light` / `themes_dark`);
- выдели layout regions: header, sidebar, content, modal, table, footer;
- определи, какие regions собираются DS-компонентами, а какие page-level layout.

Для каждого component instance:

- проверь Code Connect / docs / Git link;
- сними `componentProperties` и `variantProperties`;
- сопоставь Figma property с public prop из `borrom-ds-test`;
- не угадывай props по внешнему виду, если Figma отдала явное значение;
- проверь slot composition: label/inline, iconLeft/iconRight, actionSlot, accountMenu, bottomSlot, children.

Для auto-layout:

- сохрани direction;
- сохрани padding по всем сторонам;
- сохрани gap;
- сохрани alignment;
- сохрани fill/hug/fixed;
- не заменяй fill/hug произвольными фиксированными размерами.

Для spacing:

- не выбирай `gap-*` на глаз;
- сразу сопоставляй Figma px с token/class;
- пример: `40px -> --generic-spacing-15 -> gap-15`, а не визуально похожий `gap-10`.

Для text:

- используй токены typography;
- проверь `Roboto Flex`;
- проверь переносы, line-height и container width;
- не оставляй системный fallback-шрифт как финальный parity.

Для assets:

- используй image asset только если в Figma это реальный image/media content;
- не используй screenshot всего блока как способ сверстать блок;
- object-fit/crop бери из ноды, не подбирай глазами.

# Компоненты, Code Connect и Git links

Сначала всегда ищем готовый компонент в `borrom-ds-test`.

Если Figma-компонент содержит Git link, Code Connect или ссылку на docs, это основной источник. В таком случае нельзя верстать компонент заново по слоям, пока не проверены существующие исходники и public API.

## Порядок работы с Figma-компонентом

1. Определи компонент и его Figma instance.
2. Проверь Code Connect / docs / Git link.
3. Проверь runtime-компонент, types, styles, local export и story в дизайн-системе, если работа идет внутри DS.
4. Проверь public import из `borrom-ds-test`.
5. Сопоставь Figma variants/properties со props из types.
6. Используй готовый компонент в page-level композиции.
7. Если нужного prop/state/token нет, зафиксируй `Temporary mapping` или `Assumption`.

## Imports

```tsx
import { Button, Input, AppLayout } from "borrom-ds-test";
import "borrom-ds-test/styles.css";
```

Не импортируй из внутренних путей `src/components/...` в проекте реализации макета.

## Exact props rule

Для каждого Figma component instance переноси точные `componentProperties`, `variantProperties`, swaps и text properties в runtime props.

Не выбирай `size`, `type`, `color`, `iconOnly`, `selected`, `disabled` по визуальной похожести, если Figma уже отдала явное значение.

Особенно важно для компактных компонентов:

- `Label`;
- `Counter`;
- `Button`;
- `IconButton`;
- `CheckBox`;
- `Input`;
- `TabsOverflow`;
- `Radio`;
- `Switch`.

Один неверный prop может одновременно поменять spacing, typography, icon size, radius и state behavior.

## Code Connect

Code Connect связывает опубликованный Figma component с публичным React API дизайн-системы.

Создавай `<Component>.figma.js` только если:

- компонент уже реализован в `src/components/<Component>`;
- известен Figma URL с `node-id`;
- понятны Figma properties/variants;
- можно выдать корректный snippet через public import из `borrom-ds-test`.

Если runtime-компонента нет, Code Connect не создаем и фиксируем blocker.

Файлы в дизайн-системе:

```text
src/components/<Component>/<Component>.figma.js
src/components/<Component>/<Component>.docs.md
```

Snippet импортирует только публичный API:

```js
imports: ['import { Component } from "borrom-ds-test"']
```

Проверки для mapping:

- Figma variants сопоставлены с public props из types.
- Text properties идут в `children`, `label`, `title` или другой существующий prop.
- Boolean properties сопоставлены с реальными boolean props.
- Instance swap используется только если вложенный instance имеет Code Connect.
- Несовпадения зафиксированы как `Temporary mapping`.

# Geometry map, tokens и visual parity

Перед page-level версткой каждого экрана составь короткую geometry map.

## Geometry map template

```md
# Geometry Map

Source:
- figmaFile:
- figmaNode:
- screen:
- viewport: desktop | mobile
- theme: themes_light | themes_dark

| Area | Figma px | Token variable | Class / CSS | Status | Notes |
| --- | ---: | --- | --- | --- | --- |
| Main container width | <px> | <token> | <class> | pending | |
| Header height | <px> | <token> | <class> | pending | |
| PageHeader -> next section | 40px | --generic-spacing-15 | gap-15 | pending | desktop example |
| Section vertical gap | <px> | <token> | <class> | pending | |
| Columns horizontal gap | <px> | <token> | <class> | pending | |
| Card padding | <px> | <token> | <class> | pending | |
```

Минимальный набор:

- width основного контейнера;
- высота header/content-блоков;
- вертикальные gap между основными секциями;
- горизонтальные gap между колонками;
- padding основных контейнеров;
- расстояние `PageHeader -> next section` на desktop;
- места, где визуальный “воздух” влияет на композицию.

## Tokens

- Используем те же токены, что и в Figma.
- Цвета, spacing, radius, typography и размеры идут через CSS variables / Tailwind aliases из `src/tokens` и `src/styles/global.css`.
- Raw hex/rgb/hsl, произвольные `px` и ручные font values запрещены, если есть соответствующий token.
- Если точного токена нет, выбираем ближайший semantic token и фиксируем `Temporary mapping`.

## Spacing parity

Пример:

```text
40px -> --generic-spacing-15 -> gap-15
22px -> --generic-spacing-10 -> gap-10
```

Если в Figma `40px`, нельзя заменить это на визуально похожий `gap-10`, потому что в текущей scale `gap-10 = 22px`, а `gap-15 = 40px`.

Перед сдачей выполняй:

```text
Critical spacing parity check
```

На этом checkpoint проверяются только ключевые gap/offset без остального визуального шума.

## Visual parity

Перед финалом:

- локальный экран открыт через `npm run server:start`;
- сделан screenshot локального экрана;
- локальный screenshot сравнен с Figma screenshot / PNG-контекстом;
- PNG-контекст использован только как визуальный референс;
- проверены компоненты, spacing, typography, colors, radius, states, proportions, theme;
- отдельно проверены критические gap/offset из geometry map;
- расхождения исправлены по Figma data/tokens или зафиксированы как `Temporary mapping` / `Assumption`.

Storybook показывает компонент в изолированном сценарии. Экран в приложении - это композиция компонентов, темы, контейнеров, слотов и auto-layout. Поэтому совпадение Storybook не равно совпадению всего экрана.

# Fallback и page-level решения

Если готовой runtime-реализации нет:

1. Открываем Figma node через MCP.
2. Изучаем структуру слоев, auto-layout, constraints, variants, slots, text styles и variables.
3. Снимаем реализацию по реальным node/layer properties.
4. Не используем PNG/screenshot как источник размеров, цветов или spacing.
5. Для спорных мест фиксируем `Temporary mapping` или `Assumption`.

Если отдельный внутренний node не открывается через `get_design_context`, используем комбинацию:

- root `get_design_context` для Code Connect сигналов;
- `get_metadata` для geometry и hierarchy;
- Figma Plugin API read-only для полного дерева/geometry, если root-context слишком сжат;
- ручная page-level композиция поверх DS-компонентов.

Локальная обертка допустима только для page-level composition:

- раскладка;
- связка данных;
- адаптация слотов;
- layout recipe;
- fallback для конкретной экранной структуры.

Не создавай публичный UI-компонент, если задача решается page-level композициями и существующими DS-компонентами.

# Типовые решения из прошлых отчетов

## DS-компонент не меняем ради экрана

Если компонент корректно работает в playground/Storybook, проблему ищем в:

- page-level composition;
- props;
- slot composition;
- theme;
- scroll container;
- `overflow-hidden/auto`;
- fixed height;
- z-index;
- props вроде `maxHeight`, `side`, `align`, `contentClassName`.

Компонент дизайн-системы не меняем без отдельной задачи на DS.

## Label и компактные компоненты

`Label` и похожие компактные компоненты реализуем по явным Figma variants/properties.

Пример: если Figma показывает `Type=Flat`, `Color=Danger`, `Size=Md`, `iconOnly=Off`, то в коде используем `type="flat"`, `color="danger"`, `size="md"` и нужный icon prop, даже если `sm` визуально кажется близким.

## HeaderInside

Если в макете справа notification + avatar, используем штатный action/account контур компонента:

- `showNotification`;
- `accountMenu`;
- documented action slot.

Не подменяем правую часть произвольным `children`, если задача - 1:1 совпадение с инстансом DS.

## Table и сложные screen-specific композиции

Если DS-компонент есть, но его готовая композиция не совпадает с экранной структурой Figma, не переписываем компонент.

Решение: page-level fallback по Figma layers, сохраняя DS controls и tokens.

Пример: таблицу можно собрать как page-level grid/table, если текущий `Table` добавляет toolbar/top-slot spacing, которого нет в макете.

## Radio / checkbox / input-like

Проверяем режим композиции:

- field label через prop;
- inline-композиция `control + text`;
- helper/error text;
- selected/disabled/error state.

Эти режимы дают разную geometry и не взаимозаменяемы.

## Image/upload presets

Если DS preset отличается от конкретной Figma-ноды по пропорциям, не меняем компонент глобально. Фиксируем page-level `Temporary mapping`.

Пример: прямоугольная upload-зона поверх `InputImg`, если DS preset квадратный.

## Static mock data

Static mock data допустима только когда задача просит реализовать видимый экран, а не подключить реальные данные. Это фиксируется как `Assumption`.

## Encoding

- Все `.md`, `.js`, `.ts`, `.tsx`, `.json`, `.css` файлы сохраняются в UTF-8.
- Документационные `.md` с русским текстом в Windows-окружении сохраняем как UTF-8 with BOM.
- `.editorconfig` должен закреплять для `*.md`: `charset = utf-8-bom`.
- Перед завершением запускай `npm run encoding:check`.
- Если видны mojibake-маркеры, файл считается битым и должен быть переписан нормальным UTF-8.
